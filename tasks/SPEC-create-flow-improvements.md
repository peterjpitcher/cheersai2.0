# SPEC: Create Flow Improvements and Application Audit

**Date:** 2026-05-21  
**Status:** Draft - rewritten after code, database, cron, and CI audit  
**Complexity:** XL  
**Primary scope:** the new `CreateWizard` flow in `src/features/create`, plus the publish, database, and cron contracts it depends on.

---

## Executive Summary

The original draft correctly identified several create-flow UX problems, but it understated the more serious data and publishing risks around the wizard. The UX fixes should still happen, but they must sit behind publish-path correctness.

Highest-risk facts verified in this workspace:

1. `createScheduledBatch()` creates `content_items` and `content_variants`, but in `mode: "schedule"` it does **not** create `publish_jobs`. Those scheduled posts can appear in Planner while having no job for `/api/cron/publish-scheduler` to promote.
2. The generated review UI shows body, hashtags, CTA, link-in-bio text, and GBP CTA action, but `createScheduledBatch()` persists only `platformCopy.body` into `content_variants.body`. Reviewed hashtags and CTA fields are dropped before publishing.
3. The active QStash publish path builds provider payloads from raw `media_assets.storage_path` values, not signed/public URLs, and it does not apply banner rendering. The legacy Supabase `publish-queue` worker contains banner rendering, but Vercel cron now targets the QStash scheduler.
4. `getCalendarItemsAction()`, `getDraft()`, and `listDrafts()` use the service-role client returned by `requireAuthContext()` but omit explicit `account_id` filters. RLS does not protect those queries because service-role bypasses RLS.
5. `attachMediaToContent()` replaces attachments for any supplied `content_item_id` without first proving that the content item and media assets belong to the current account.
6. The schedule calendar's existing-item preview path reads `media_library.url`, but the actual compatibility migration stores `media_library.file_url` as a storage path, not a signed URL. Existing calendar thumbnails are therefore not reliably renderable.
7. Cron route auth is duplicated, direct string comparison is used, and most handlers accept `?secret=...` despite Vercel sending `CRON_SECRET` as a `Bearer` `Authorization` header.

This spec therefore has two workstreams:

- **Part A:** create-flow correctness and UX improvements.
- **Part B:** application audit findings that must be addressed or explicitly deferred before the create flow can be called production-safe.

## Verified Current Architecture

### Create Wizard

The new wizard is `src/features/create/create-wizard.tsx`:

- It is a 4-step client flow: Brief -> Media -> Schedule -> Generate.
- It creates an initial draft via `createDraft()`.
- It loads media with `getCreateModalData()`, which calls `listMediaAssets()` and returns signed `previewUrl` values for the media picker.
- It attaches selected media on the Media -> Schedule transition through `attachMediaToContent()`.
- It stores wizard state in `content_items.body_draft` through `saveDraft()` and `useAutoSaveDraft()`.
- It generates one `SlotGeneratedCopy` per selected schedule slot.
- It schedules or queues all ready generated slot copies through `createScheduledBatch()`.

### Scheduling

The schedule step is `src/features/create/steps/schedule-step.tsx` and the calendar UI is `src/features/create/schedule/schedule-calendar.tsx`.

Verified behavior:

- Suggestions are generated by content type:
  - Events use `buildEventSuggestions()` -> `buildEventCadenceSlots()`.
  - Promotions use `buildPromotionSuggestions()`.
  - Weekly recurring uses `buildWeeklySuggestions()`.
- Existing items are loaded by dynamically importing `getCalendarItemsAction()`.
- Fetch failures currently degrade to an empty list and are not surfaced to the user.
- Manual slot creation currently opens an inline `<input type="time">` plus Add/Cancel buttons.
- Suggestion clicks attach labels because `handleAddSlot()` matches exact date+time against `suggestions`.
- Manual custom slots get `label: undefined` when no exact suggestion matches, even if the chosen date is the event date or promotion end date.
- Story slot limit is 1; other content types are capped at 12.

### Generation

The generate step is `src/features/create/steps/generate-step.tsx`.

Verified behavior:

- `generateContent()` and `regenerateWithModifier()` receive `mediaIds`, `scheduledAt`, and `slotLabel`.
- `buildUserPrompt()` includes schedule time and a generic `Slot purpose: "..."`

- Body textareas are fixed at `rows={4}` and `resize-none`.
- No image preview or `BannerOverlay` is rendered in the generated review state.

### Final Batch Persistence

The final wizard write path is `createScheduledBatch()` in `src/app/actions/content.ts`.

Verified behavior:

- For event, promotion, and weekly content, one `campaigns` row is created.
- For every slot x platform, one `content_items` row is inserted.
- For every content item, one `content_variants` row is upserted.
- `content_variants.body` is set to only `slot.copy[platform].body`.
- `content_variants.preview_data` exists in the schema but is not populated here.
- `content_media_attachments` rows are inserted for v2 compatibility.
- In `mode: "queue_now"`, `enqueueAndDispatch()` is called.
- In `mode: "schedule"`, no publish job is created.
- The campaign metadata shape is `{ brief, slotCount }`, which does not match the top-level timing fields expected by `extractCampaignTiming()` and the duplicated Supabase worker logic.

### Publishing and Cron

Verified local cron/publish facts:

- `vercel.json` schedules:
  - `/api/cron/publish-scheduler` every minute.
  - `/api/cron/purge-trash` daily.
  - `/api/cron/sync-meta-campaigns` daily.
  - `/api/cron/optimise-meta-campaigns` daily.

  - `/api/cron/notify-failures` hourly.

Relevant verified tables and compatibility facts:

- `token_vault` exists and is used by the newer provider adapters through `getDecryptedToken()`.

# Part A: Create Flow Correctness and UX

## A0: Create Publish Jobs for Scheduled Wizard Batches

### Problem

`createScheduledBatch()` writes future scheduled `content_items`, but only calls `enqueueAndDispatch()` when `mode === "queue_now"`. The Vercel scheduler only promotes rows in `publish_jobs`; it does not scan `content_items` directly.

### Required Behavior

Every inserted content item must get a matching `publish_jobs` row.

- For `mode: "schedule"`: create publish jobs with `status: "scheduled"` and `scheduled_at = slot.scheduledAt`. Do not dispatch immediately unless the scheduled time is inside the existing immediate threshold.
- For `mode: "queue_now"`: keep current immediate dispatch behavior.
- Ensure one job per content item/platform/scheduled time.
- If job creation fails after content rows are inserted, return a hard error and do not silently leave orphan scheduled content.

### Implementation Notes

Prefer using `enqueueAndDispatch()` for both modes because it already creates `publish_jobs` and only dispatches near-immediate jobs. If partial failure handling becomes too fragile in server-action code, move the batch insert + variant insert + attachment insert + publish job insert into a database RPC transaction.

### Files

| File | Change |
|---|---|
| `src/app/actions/content.ts` | Call `enqueueAndDispatch()` for every inserted item in both `schedule` and `queue_now` modes |
| `src/lib/publishing/queue.ts` | Keep existing immediate-threshold behavior; add tests if needed |
| `tests/app/actions/content.test.ts` or new focused test | Assert scheduled mode creates publish jobs |

### Acceptance Criteria

- Scheduling 3 slots to 3 platforms creates 9 `content_items`, 9 `content_variants`, and 9 `publish_jobs`.
- Future scheduled jobs remain `scheduled` and are not immediately dispatched.
- Due/near-due jobs are dispatched through QStash.

## A2: Write Campaign Metadata in the Shape Planner and Banner Code Expect

### Problem

`createScheduledBatch()` writes campaign metadata as `{ brief, slotCount }`. Planner and publish-time banner logic call `extractCampaignTiming()`, which expects top-level fields such as:

- `eventStart` or `startDate` plus `startTime`
- `endDate` for promotions
- `dayOfWeek` and `time` for weekly campaigns

With the nested `brief` shape, planner and publish-time code cannot reliably compute proximity labels such as `TODAY`, `TONIGHT`, `ENDS TOMORROW`, or `THIS FRIDAY`.

### Required Behavior

When `createScheduledBatch()` creates a campaign, write the same timing metadata contract used by `src/lib/create/service.ts`.

Event metadata:

```ts
{
  eventStart: DateTime.fromISO(`${brief.eventDate}T${brief.eventTime}`, { zone: DEFAULT_TIMEZONE }).toISO(),
  startDate: brief.eventDate,
  startTime: brief.eventTime,
  endDate: brief.eventEndDate ?? null,
  brief,
  slotCount,
}
```

Promotion metadata:

```ts
{
  startDate: brief.startDate ?? null,
  endDate: brief.endDate,
  offerSummary: brief.offerSummary,
  couponCode: brief.couponCode ?? null,
  brief,
  slotCount,
}
```

Weekly metadata:

```ts
{
  dayOfWeek: brief.dayOfWeek,
  time: brief.time,
  weeksAhead: brief.weeksAhead,
  brief,
  slotCount,
}
```

### Files

| File | Change |
|---|---|
| `src/app/actions/content.ts` | Add a metadata builder for `createScheduledBatch()` |
| `src/lib/scheduling/campaign-timing.ts` | Optionally support nested `brief` as a defensive fallback |
| `supabase/functions/publish-queue/banner-label.ts` | Keep duplicated extraction logic in sync if fallback is added |
| Tests | Add metadata contract tests for event, promotion, weekly |

### Acceptance Criteria

- Planner banner labels for wizard-created event/promotion/weekly campaigns match labels from the older campaign creation paths.
- Publish-time banner logic can compute the same labels.

# Part B: Application Audit Findings

## P0 / P1 Security and Publish Correctness

### B1: Unauthenticated Supabase Edge Functions

`supabase/config.toml` sets `verify_jwt = false` for:

- `publish-queue`
- `materialise-weekly`
- `media-derivatives`

Their handlers accept `POST` and can execute service-role work without checking `CRON_SECRET`.

Fix:

- If these functions are retired, remove or disable them.
- If still callable, add a timing-safe shared-secret check before JSON parsing or database work.

### B2: Service-Role Server Actions Need Explicit Account Guards

`requireAuthContext()` returns a service-role Supabase client. That bypasses RLS.

Confirmed missing/weak guards:

- `getDraft()` omits `.eq("account_id", accountId)`.
- `listDrafts()` omits `.eq("account_id", accountId)`.
- `getCalendarItemsAction()` omits `.eq("account_id", accountId)`.
- `attachMediaToContent()` does not validate ownership of the content item or supplied media assets.
- `reorderLinkInBioTiles()` upserts caller-supplied ids with `onConflict: "id"` without first proving every tile belongs to the account.

Fix:

- Add account filters to every service-role query.
- Validate all ids before mutation.
- Reject partial id sets rather than silently dropping unknown ids.

### B3: Active QStash Publish Payload Is Incomplete

The QStash path in `src/lib/publishing/handler.ts` currently:

- Reads raw `media_assets.storage_path` and passes those values to provider APIs as `mediaUrls`.
- Does not sign media URLs.
- Does not render banner overlays.
- Does not populate `eventDetails` or `offerDetails` for GBP event/offer publishing.
- Does not call `adapter.validate()` before publishing.

The legacy Supabase `publish-queue` worker has richer banner handling, but `vercel.json` and `/api/cron/publish` indicate the active scheduler path is QStash.

Fix:

- Resolve publish media to provider-accessible signed/public URLs.
- Apply banner rendering before provider upload when banner config requires it.
- Build `ContentPayload.eventDetails` and `ContentPayload.offerDetails` from campaign metadata or prompt context.
- Call adapter validation and fail with classified errors before attempting provider API calls.

### B4: Token Vault Migration Is Incomplete

The new connection flow stores OAuth tokens in `token_vault`, but old paths still read plaintext `social_connections.access_token` and `refresh_token`, including:

- `src/lib/publishing/preflight.ts`
- `supabase/functions/publish-queue/worker.ts`
- review/cron paths that refresh GBP reviews

Fix:

- Route all provider-token reads through `getDecryptedToken()` or a shared token access layer.
- Backfill legacy plaintext tokens into `token_vault`.
- Only after verification, remove plaintext token columns and old code paths.

### B5: Cron Auth Is Duplicated, Non-Timing-Safe, and Accepts URL Secrets

Most cron routes manually compare secrets and accept `?secret=`. Vercel sends `CRON_SECRET` as a `Bearer` authorization header when configured.

Fix:

- Add a shared `src/lib/security/cron-auth.ts`.
- Use `validateSecret()` from `src/lib/security/signing.ts`.
- Accept `Authorization: Bearer <CRON_SECRET>` and optionally `x-cron-secret` for local/manual ops.
- Remove URL query-secret support unless there is a documented external caller that cannot set headers.
- Preserve `GET` handlers for all Vercel-scheduled routes.

### B6: QStash Failure Callback Exists but Is Not Wired

`/api/webhooks/qstash-publish/failure` exists, but `dispatchToQStash()` does not set `failureCallback`.

Fix:

- Set QStash `failureCallback` to `${NEXT_PUBLIC_SITE_URL}/api/webhooks/qstash-publish/failure`.
- Add a dispatch unit test.
- Confirm the failure callback payload contains the `jobId` shape expected by the handler.

### B7: Dependency Audit Fails

Current `npm audit --audit-level=moderate` reports 9 vulnerabilities, including high-severity advisories for `next`, `flatted`, `minimatch`, `picomatch`, and `tar`.

Fix:

- Patch `next` and `eslint-config-next` together.
- Run full regression after dependency updates.
- Add a high-severity audit gate to CI once the current backlog is cleared.

### B12: `proxy.ts` Is in the Wrong Location

`src/app/proxy.ts` contains the intended Next 16 proxy auth guard. Next expects proxy at the project root or `src/proxy.ts`, not inside `src/app`.

Current root `middleware.ts` only handles apex-to-www redirects. Protected layouts still enforce auth, but the intended proxy layer is not active.

Fix:

- Move `src/app/proxy.ts` to `src/proxy.ts`.
- Keep or merge the apex redirect behavior.
- Verify public routes and API routes are not accidentally protected.

### B13: Public SEO Is Globally Disabled

`robots.txt` disallows everything, root metadata sets `noindex`, and security headers set `X-Robots-Tag: noindex, nofollow` globally.

This is acceptable for the private app but wrong if public link-in-bio pages should index or preview normally.

Fix:

- Split private-app noindex policy from public `/(public)/l/[slug]` behavior.

### B14: Production CSP Is Permissive

`src/lib/security/headers.ts` includes both `unsafe-eval` and `unsafe-inline` in `script-src`.

Fix:

### B15: Link-in-Bio Page View Tracking Is Unreliable with ISR

# Deployment Safety

No schema migration is required for Part A if `preview_data`, existing `banner_*`, and existing `publish_jobs` columns are used.

Migrations or explicit production DB approvals are required for:

- Dropping unused views.
- Dropping plaintext token columns.
- Any RPC transaction introduced for batch creation.

Operational cautions:

- Do not deploy A0 without B3 if the newly created publish jobs would flow into an incomplete QStash payload path.
- Do not remove URL cron secrets until all external/manual callers are confirmed to use headers.
- Do not disable Supabase Edge Functions until production scheduling/publishing has been confirmed to use the Next.js/QStash path only.
- Do not drop `media_library` while `content_media_attachments` still references it.
