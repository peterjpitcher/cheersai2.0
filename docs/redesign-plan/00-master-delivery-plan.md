# CheersAI Redesign — Master Delivery Plan

> Generated by 8-agent parallel review team. Each section links to the full specialist document.

---

## Build Approach — DECIDED (2026-03-05)

**Greenfield rebuild in a new repository.** The current `cheersai2.0` repo is reference and audit material only. No code is carried forward. The rebuild starts from a fresh Next.js project, new Supabase project, and new git repository. This eliminates all inherited technical debt, incorrect conventions, and structural anti-patterns identified in this audit.

The redesign plan documents in this folder are the specification for the new build.

---

## Executive Summary

The current CheersAI 2.0 codebase is a working prototype with a sound product vision but significant structural debt that makes it unsafe to build on at production scale. The review team identified **6 Critical issues**, **28 High/Major issues**, and **30+ Minor issues** across all disciplines.

The rebuild is warranted. This plan organises it into four phases with clear exit criteria, sequenced to unlock business value as early as possible while eliminating the most dangerous technical risks first.

## Target Architecture Summary

### Stack (unchanged from current, corrected usage)

| Layer | Technology | Changes |
|-------|-----------|---------|
| Frontend | Next.js 16 App Router | Remove `--webpack`, fix RSC/client boundaries |
| Styling | Tailwind v4 + Radix UI | Consolidate component library, remove dead components |
| Data fetching | Server Components + TanStack Query | Server Components for initial data, TQ for mutations/live |
| Auth | Supabase Auth + SSR | Add middleware guard, AES-256-GCM token encryption |
| Database | Supabase Postgres | New schema with idempotency, junction tables, audit log |
| Storage | Supabase Storage | Add `media_renditions` table, use `next/image` everywhere |
| Background jobs | QStash (replace Vercel Cron) | Signed delivery, built-in retry, message IDs as idempotency keys |
| AI | OpenAI GPT-4o | Add timeouts, expose controls in UI |
| Email | Resend | Retain |
| Observability | Axiom (add) | Structured JSON logs, metrics, alerts |

## Phased Rebuild Plan

### Phase 1 — Foundations & Security (Weeks 1–3)

**Goal:** Safe, deployable skeleton with all critical security issues resolved.

**Deliverables:**
- New Next.js project with correct App Router structure, Turbopack enabled
- Middleware auth guard on all `(app)/*` routes
- AES-256-GCM token encryption helper + migration to encrypt existing tokens
- GBP refresh token flow implemented
- QStash replacing Vercel Cron for publish jobs
- Idempotency key on `publish_jobs` with unique constraint
- Structured logging with Axiom (correlation IDs, job IDs, durations)
- Consolidated schema baseline migration
- `content_media_attachments` junction table replacing `media_ids uuid[]`
- Security headers (CSP, HSTS, X-Frame-Options, Referrer-Policy)
- OAuth callback session validation + HMAC state verification

**Exit criteria:**
- [ ] All C-1 through C-6 critical issues resolved
- [ ] Auth middleware test: unauthenticated request to any `(app)/*` route → 302 to login
- [ ] Cron endpoint test: request without valid `CRON_SECRET` → 401
- [ ] Token encryption test: `social_connections.access_token` column contains ciphertext, not plaintext
- [ ] Double-fire test: two simultaneous publish jobs for same content item → second is a no-op
- [ ] GBP test: publish job runs 2h after token issue → refresh token used, post succeeds

### Phase 2 — Content Workflows (Weeks 4–6)

**Goal:** Owner can create, generate AI copy, and schedule content for all three platforms.

**Deliverables:**
- Unified Create flow: Instant Post, Event Campaign, Promotion Campaign, Weekly Recurring
- GBP Event and Offer post types implemented in all forms
- Platform-specific editor with per-tab previews (Facebook, Instagram, GBP)
- AI generation with timeout (30s), exposed tone/length controls, Regenerate with modifier
- Conflict detection surfaced in scheduling UI with resolution suggestions
- Bulk approve for campaign content items
- Pre-flight error messages in plain English with actionable steps
- Media library with search, tagging, and campaign filters
- `next/image` replacing all bare `<img>` tags
- Mobile bottom nav (5 items)

**Exit criteria:**
- [ ] Owner can create an Event campaign, generate AI copy for all platforms, and schedule in ≤15 minutes
- [ ] GBP Event post creates correctly on Google listing
- [ ] GBP Offer post creates correctly with coupon code
- [ ] Conflict: scheduling two posts on same platform within 10 minutes shows warning
- [ ] Pre-flight failure: expired token → plain-English message with "Reconnect" CTA
- [ ] AI generation timeout: 30s exceeded → graceful error, not hanging spinner
- [ ] Mobile: all touch targets ≥44×44px

### Phase 3 — Publishing Pipeline (Weeks 7–9)

**Goal:** Reliable publish-to-live with monitoring, failure recovery, and notifications.

**Deliverables:**
- Full publish pipeline for all content types: Facebook posts/stories/events, Instagram posts/stories, GBP updates/events/offers
- Instagram carousel support
- Retry/backoff: 5m/15m/45m, 4 attempts max, QStash-native retry
- Publish failure recovery: "Retry now" button + "Download copy & assets" fallback package
- Token health monitoring: proactive refresh 5 days before expiry, in-app + email notification
- Planner failure banner (Attention Needed count at top of view)
- Activity feed with real-time updates (Supabase Realtime subscription)
- Email alerts for publish failures and token expiry (Resend)
- Publish job audit log (`audit_log` table)
- `content_item_versions` snapshot at publish time

**Exit criteria:**
- [ ] Publish success rate ≥99% on staging with mock providers
- [ ] Failure recovery: after 3 failed attempts → "Download assets" link appears in planner detail
- [ ] Token expiry: token expiring in 4 days → in-app notification + email sent
- [ ] Double-publish: simulated Cron double-fire → only one publish attempt executed
- [ ] Activity feed: post published → planner status updates within 5 seconds without refresh

### Phase 4 — Polish, Performance & Hardening (Weeks 10–12)

**Goal:** Production-ready quality — performance budgets met, test coverage thresholds hit, go-live.

**Deliverables:**
- Performance: LCP <2.5s on Planner, INP <200ms for all interactions
- Test coverage thresholds enforced: scheduling ≥90%, publishing pipeline ≥85%, auth ≥80%
- MSW integration test suite for all provider API flows
- Playwright E2E suite covering 6 critical journeys
- Full design system consolidation (single Sidebar, single modal pattern, design tokens)
- WCAG 2.1 AA compliance: contrast ratios, touch targets, keyboard nav, ARIA
- Timezone support enabled and tested
- Staging environment with mock providers for full regression
- Runbooks: token reconnection, publish outage, credential rotation
- Go-live checklist completion

**Exit criteria:**
- [ ] Lighthouse scores: Performance ≥85, Accessibility ≥95 on all primary routes
- [ ] Test suite: all CI gates green, coverage thresholds met
- [ ] Load test: 50 concurrent requests to Planner → p99 <500ms
- [ ] WCAG audit: zero AA failures on all primary routes
- [ ] Go-live checklist: all items checked off by owner and engineer

## Dependencies and Sequencing

```
Phase 1 (Security foundations)
  └── Phase 2 (Content workflows) — requires: schema stable, auth secure
        └── Phase 3 (Publish pipeline) — requires: content model complete, integrations fixed
              └── Phase 4 (Polish) — requires: all features shipped, test infrastructure ready
```

**Cross-phase dependencies:**
- QStash setup (Phase 1) is required before any background job work in Phase 3
- `content_media_attachments` junction table (Phase 1) is required before media attachment UI in Phase 2
- Token encryption (Phase 1) is required before any OAuth work in Phase 2 or 3
- Structured logging (Phase 1) must be in place before Phase 3 (reliability cannot be verified without it)

## Risk Register

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|-----------|
| Facebook/Meta API policy change breaks publish | Medium | High | Provider abstraction layer; monitor Meta changelog; keep fallback download path |
| Instagram story programmatic publishing unsupported for account type | Medium | Medium | Add capability pre-check at connection time; surface clear error before scheduling |
| GBP API deprecation (v4 already legacy) | High | Medium | Migrate to current API version in Phase 1; track Google My Business API changelog |
| OpenAI rate limits or quality regression | Low | Medium | Add retry with backoff; allow manual copy override at all times |
| Vercel function timeout kills long publish jobs | Medium | High | Use QStash for async publish; keep publish handler <10s; use streaming where needed |
| Schema migration data loss | Low | Critical | Test migrations on staging clone before prod; take snapshot before each migration |
| Owner locks themselves out (forgot password, expired passkey) | Low | High | Ensure magic link always works as recovery; document recovery procedure |

## Open Decisions — RESOLVED (2026-03-05)

| # | Decision | Owner Decision | Engineering Implication |
|---|---------|---------------|------------------------|
| 1 | Instagram story publishing capability | **Supported — build it fully.** No capability check or fallback needed. | Stories are a first-class content type. Full implementation required in Phase 3. Remove any planned capability pre-check gate. |
| 2 | Bulk approve granularity | **Both: approve all AND select individually.** | Planner needs a "Select" mode with per-item checkboxes, plus "Approve selected" and "Approve all" actions. |
| 3 | Activity feed refresh | **Live via Supabase Realtime.** | Subscribe to `content_items` and `publish_jobs` table changes in the Planner. No polling fallback needed. |
| 4 | Publish failure fallback | **Fix the problem — no download ZIP.** | Remove download-assets from scope entirely. Invest instead in better retry UI, plain-English root-cause display, and Reconnect CTA. |
| 5 | Weekly recurring auto-publish | **Auto-publish once approved.** Approval is a permanent authorisation for that slot. | Set `auto_confirm = true` at campaign level on first approval. Weekly materialiser publishes without re-confirmation each week. |
| 6 | Analytics in v1 | **Yes — build analytics in v1** to drive content improvement. | Add analytics module to Phase 4 scope. Minimum: per-post publish outcome, platform breakdown, weekly summary view. Add `analytics_snapshots` table to schema. Scope as sixth nav item or Planner tab. |
| 7 | Notification channels | **In-app for non-urgent, email for urgent.** | Urgency tiers: Urgent (publish failure, token expired/disconnected) = email + in-app. Non-urgent (token expiring soon, weekly summary) = in-app only. Add `urgency` enum column (`urgent` / `standard`) to `notifications` table. |
| 8 | Timezone | **Europe/London only — hardcoded, no UI.** | Remove timezone field from Settings entirely. Hardcode `Europe/London` in all scheduling logic. Simplifies the scheduling engine; no DST UI edge cases to handle. |
