## Key Integrations

### OpenAI
AI-powered content generation and streaming. Used in the Create flow (`/api/create/generate-stream`), campaign actions, review reply generation, and the shared AI client at `src/lib/ai/client.ts`.

### Resend
Transactional email for failure notifications and expiring connection alerts. Core client at `src/lib/email/resend.ts`, consumed by cron jobs.

### Meta (Facebook / Instagram)
Social media publishing, OAuth connection management, and ad campaign optimisation. Includes dedicated OAuth callbacks, ad account management, and campaign sync/optimisation cron jobs.

### Google My Business
Review syncing and reply management. Connected via OAuth with a dedicated cron job for syncing GBP reviews.

### Supabase
PostgreSQL database with Row Level Security. Two client patterns: cookie-based auth client for user operations, service-role client for system/cron operations. See [[relationships]] for the full table-to-action map.

## Auth Model

Authentication uses Supabase Auth with JWT tokens stored in HTTP-only cookies.

**Middleware** (`middleware.ts`): Only handles apex-to-www domain redirect (`cheersai.uk` to `www.cheersai.uk`). Does not enforce auth -- that happens at the layout level.

**Auth flow**:
1. `middleware.ts` -- domain redirect only
2. `(app)/layout.tsx` -- calls `getCurrentUser()` which calls `requireAuthContext()`
3. `requireAuthContext()` -- verifies JWT via `supabase.auth.getUser()`, resolves account ID, redirects to `/login` if unauthenticated
4. Server actions -- re-verify auth via `requireAuthContext()` on every mutation
5. Cron/API routes -- verify `CRON_SECRET` header for system operations

**Auth library** (`src/lib/auth/`):
- `server.ts` -- `requireAuthContext()`, `getCurrentUser()`
- `actions.ts` -- `signOut()`
- `rate-limit.ts` -- IP-based rate limiting with in-memory fallback
- `types.ts` -- `AppUser` interface

See [[routes]] for per-route auth type details and [[relationships]] for cross-references.
