## Auth Model

CheersAI enforces auth at the **layout level**, not in middleware. The root `middleware.ts` only handles apex→www host redirects.

| Layer | Behaviour |
|-------|-----------|
| `middleware.ts` (root) | Redirects `cheersai.uk` → `www.cheersai.uk` (308). **No** auth check. |
| `src/app/(app)/layout.tsx` | Calls `getCurrentUser()` — redirects to `/login` if unauthenticated; wraps children in `AuthProvider` + `AppShell`. |
| `src/app/(auth)/layout.tsx` | Reverse-guard: if user *is* signed in, redirects to `/planner`. |
| `src/app/(public)/layout.tsx` | No auth — serves public link-in-bio pages. |
| Server actions | Every action calls `requireAuthContext()` to re-verify server-side. |
| `/api/auth/*` | Rate-limited (`isRateLimited`) before Supabase login. |
| `/api/cron/*` | Verify `CRON_SECRET` via `x-cron-secret` header or `Authorization: Bearer`. |
| `/api/internal/render-banner` | Uses `INTERNAL_RENDER_SECRET`. |
| `/api/create/generate-stream` | Calls `supabase.auth.getUser()` directly. |

Service-role client (`createServiceSupabaseClient` at `src/lib/supabase/service.ts`) is used only for system operations that must bypass RLS.
