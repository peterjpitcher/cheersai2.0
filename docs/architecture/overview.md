## Auth Model

Authentication uses Supabase Auth with JWT tokens stored in HTTP-only cookies. The flow is:

1. **Middleware** (`middleware.ts`) -- handles apex-to-www domain redirect only; does not enforce auth
2. **Layout-level auth** -- the `(app)` route group layout calls `getCurrentUser()` from `src/lib/auth/server.ts`, which calls `requireAuthContext()`. If no valid session exists, the user is redirected to `/login`
3. **Auth layout** (`(auth)/layout.tsx`) -- checks if user is already authenticated; if so, redirects to `/planner`
4. **Server actions** -- all mutations call `requireAuthContext()` which re-verifies the Supabase session server-side
5. **Cron routes** -- authenticate via `CRON_SECRET` header comparison (not user sessions)
6. **API key routes** -- the feed API uses per-tournament `feed_api_key`; booking conversions use `BOOKING_CONVERSION_INGEST_SECRET`
7. **Internal routes** -- `render-banner` uses `CRON_SECRET` for internal service-to-service auth

Key auth files:
- `src/lib/auth/server.ts` -- `requireAuthContext()`, `getCurrentUser()`
- `src/lib/auth/rate-limit.ts` -- rate limiting with `auth_rate_limits` table
- `src/lib/auth/types.ts` -- `AppUser` type definition
- `src/lib/supabase/server.ts` -- `createServerSupabaseClient()`
- `src/lib/supabase/service.ts` -- `createServiceSupabaseClient()` (admin)
